# CANN 工具箱同学使用说明

这份说明给第一次拿到工具箱的同学用。目标不是把所有 CANN 命令讲完，而是先能连上板子、按顺序跑一个新算子，并知道哪些步骤可以点按钮，哪些步骤目前还需要手动补文件。

## 1. 启动前检查

### 本机需要什么

- Python 3.10 或更新版本。
- Python 包：`paramiko`。本地分析插件还可能需要 `openpyxl`、`matplotlib`、`numpy`。
- 整个项目目录最好保持原结构一起拷贝，不要只拷 `cann_toolbox` 一个文件夹。部分插件脚本放在 `官方算子开发工具/` 下，工具箱会用项目根目录自动定位。

启动命令：

```powershell
python .\cann_toolbox\run_toolbox.py
```

如果这台电脑的 `python` 命令不可用，就改用自己的 Python 完整路径，例如：

```powershell
"C:\Users\lv\AppData\Local\Programs\Python\Python314\python.exe" .\cann_toolbox\run_toolbox.py
```

### 板子连接

默认配置：

- IP：`192.168.0.2`
- 端口：`22`
- 用户：`HwHiAiUser`
- CANN 路径：`/usr/local/Ascend/cann-8.5.0`
- 默认工作目录：`/home/HwHiAiUser/work`

如果工具箱连不上板子，先看 Windows 的 USB RNDIS 网卡。正常情况下，本机 USB 网卡应设为：

```text
192.168.0.1 / 255.255.255.0
不要填网关
```

板端是：

```text
192.168.0.2 / 24
```

## 2. 跑一个新算子的推荐顺序

可以把它想成“新手教学模式”的关卡路线。现在工具箱还没有自动高亮下一步，但左侧功能已经尽量按这条顺序排。

1. **msKPP 理论构建**
   用来先估算切分策略、访存量、理论瓶颈。它适合看趋势，不等于板端真实性能结论。

2. **msOpGen 生成算子描述 JSON**
   用 `【插件】算子描述 JSON 生成器（msOpGen 第一步）` 生成描述文件。保存到本地后，用文件管理器传到板子。

3. **msOpGen 生成工程**
   用“生成算子工程（gen）”。重点检查：
   - 算子描述 JSON 路径。
   - 框架类型。
   - 310B 板端默认用 `ai_core-ascend310b`。

4. **写或替换 kernel / host 代码**
   这一步目前主要靠手动编辑工程文件。工具箱可以打开板端文件管理器，但还没有完整 VS Code 级编辑体验。

5. **构建 OPP 工程**
   普通安装验证用 Release 构建。
   如果后面要做 `msOpGen sim -reloc`、msDebug 或 msSanitizer 插桩实验，再用 Debug/特殊编译选项路线。

6. **安装自定义 OPP 包**
   安装目录建议每个算子/每种模式分开，例如：
   - `/home/HwHiAiUser/custom_opp_rgb_release`
   - `/home/HwHiAiUser/custom_opp_rgb_debug`
   - `/home/HwHiAiUser/custom_opp_san_bug`

7. **msOpST 生成测试用例骨架**
   用 `msopst create` 生成测试工程。

8. **准备 msOpST case JSON**
   用 `【插件】msOpST case JSON 生成器（run 前补测试配置）` 生成测试配置。默认是 AddCustom 的 `x/y -> z`、`float16`、`ND`、`shape=[2048]`，换新算子时改算子名、case 名、输入输出名字和 shape。

9. **msOpST run 功能测试**
   用“运行算子自测（run）”。如果结果通过，说明功能正确性基本过关。

   左侧还有一个 `ascendc_test 生成（仅探路）`，它不是这条主线的一步。它生成的是 kernel 直跑/调试脚手架，不做 golden 自动比对；而且对 kernel 参数和 C++ 排版很挑，`blockDim` 也写死为 1，不能用来证明切分策略。第一次跑新算子时先不要点它。

10. **msProf 上板采集**
    用 op 上板采集或传统 db 采集。它回答“真实板端跑得怎么样”。

11. **msProf / 本地分析插件整理结果**
    可用 CSV 转 Excel、Roofline、PIPE 堆叠图、Cache/带宽等插件，把采集结果转成图表和表格。

12. **msOpGen 软件仿真生成 dump**
    用于拿指令流水图。注意它是软件仿真，适合看指令流水和趋势，不代表 310B4 真实性能。

13. **msOpGen sim 转时间线**
    用 dump 生成 Chrome tracing JSON，再用 `【插件】dump2trace JSON 转 Excel` 做表格摘要。

14. **msSanitizer 异常检测**
    用来查越界、未初始化、竞争、同步问题，但这里还在探索。工具箱只是按官方功能给了命令行入口，不保证当前算子已经能被稳定检测。我们已确认 310B 上工具能启动；下一步还要继续验证怎样让 `-g/-sanitizer` 插桩真正进入 msOpGen binary kernel。所以如果它没报错，不能直接当作“算子一定没问题”。

## 3. 当前还缺的插件/向导

优先级最高：

- **新手教学模式**：按“当前算子进度”高亮下一步，告诉用户应该点哪个工具、需要填哪些路径、产物会在哪里。
- **msOpST case JSON 进阶模板**：当前已有基础生成器；后续可继续补动态 shape、`shape_range`、固定输入值、期望值脚本模板等复杂场景。
- **一键环境检查**：检查板子 SSH、CANN 路径、`mssanitizer/msopgen/msopst/msprof` 是否存在，以及 Windows RNDIS IP 是否像 `192.168.0.1/24`。
- **产物定位器**：跑完一步后自动告诉用户下一步要选哪个目录或文件，例如 `.run` 包、`run_out`、`OPPROF_*`、`dump2trace_core0.json`。

可以后补：

- 板端代码编辑增强：语法高亮、查找、保存前检查。
- msSanitizer 插桩编译向导：自动切换 `-g/-sanitizer/-O0` 等编译选项，并提醒不要污染 Release 包。
- 板端清理助手：按时间、大小、工具类型整理旧实验目录。

## 4. 使用时的安全习惯

- 不确定目录用途时，先用文件管理器看修改时间和大小，不要直接删。
- 自定义 OPP 安装目录尽量分开，不要把不同算子的包都装到同一个目录。
- Debug、Sanitizer、故意注入 bug 的工程不要拿来给同学做功能验证；异常检测按钮目前是探索入口，不是稳定验收步骤。
- 跑长命令前先看“生成的命令”，确认路径是自己的算子目录。
